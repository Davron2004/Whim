# Bug Report — 2026-06-13

Exploratory session via ADB on Android emulator (Pixel 9 Pro XL, Android 16.0, release APK).

---

## B1 — Water Counter card always opens Tip Splitter [CRITICAL]

**Symptom:** Tapping the Water Counter card on the home screen consistently renders Tip Splitter instead. Confirmed via:
- Exact card coordinates from `uiautomator dump` (`bounds="[474,231][864,621]"`, center 669×426).
- Debug panel `PAINT` section always shows `"appName": "Tip Splitter"` and `"generation": 1`.
- Reproduced 5+ times across different navigation paths (BACK key, Exit-to-Home button).

**Root cause (traced through code):** `MiniAppView.useEffect` calls `host.deliverBySource(record, wcBundle)` immediately on mount, before the WebView page has finished loading. The inner `injectJavaScript` call that sets `pendingSource = WC_BUNDLE` on the outer page is silently dropped (WebView not yet ready). When the outer page later loads and the iframe fires `ready`, `pendingSource` is still `null`, so the orchestrator falls through to the baked-in default (`initial: 'tip-splitter'`) and delivers Tip Splitter.

The generation counter confirms this: `generation=1` is always reported, meaning only the channel-(a) baked bundle ran; channel-(b) delivery never fired.

**Files involved:**
- `src/host/launcher/MiniAppView.tsx:28` — `useEffect` with no deps calls `deliverBySource` immediately
- `src/host/launcher/useMiniAppHost.ts` — `deliverBySource` → `control(js)` → `injectJavaScript`
- `build/assemble.mjs:95` — `initial: 'tip-splitter'` is baked into RUNTIME_HTML

## B2 — Reset button resets to wrong values

**Symptom:** Tapping Reset in Tip Splitter sets Bill to `0` and People to `1` instead of restoring the default values of 100 / 20 / 4. (Confirmed by the user.)

**Impact:** The "Reset" affordance is effectively broken — it produces a nonsensical app state (zero bill, one person) instead of the expected starting point.

## B3 — Diagnostic panel always visible in mini-app view [should not ship]

**Symptom:** The bottom ~55% of the mini-app screen is taken up by a developer diagnostic panel with three sections:
- `DELIVERY` — raw constraint log (`REJECTED unauthenticated frame (kind=spoof-probe) — forged control message ignored (constraint #4)`)
- `PAINT` — JSON payload including `generation`, `mountToFirstPaintMs`, `appName`
- `CONTAINMENT` — full probe JSON with `contained=true negCtl=true deliveryLeakCaught=true`, T7 re-injection report, and individual probe results

This UI is correctly valuable for development but should be hidden (`showDiagnostics: false`) in any user-facing build. Currently it is always shown.

**File:** `build/build.mjs` around the `buildOuterHtml` call — `showDiagnostics` is not passed or is defaulting to `true`.

## B9 — App content renders behind the system status bar

**Symptom:** On the home screen and inside mini-apps, the top of the app's own UI (e.g. the "Whim" title, the green "CONTAINED" diagnostic bar) overlaps with Android's native status bar — the row showing the clock, Wi-Fi signal, and battery level. The system icons and the app content are drawn on top of each other.

**Root cause:** Android 15+ enforces edge-to-edge mode by default, making the status bar transparent. `HomeScreen.tsx` uses `SafeAreaView` imported from `react-native` (line 15), which does not account for Android insets in edge-to-edge mode. `App.tsx` already has `SafeAreaProvider` from `react-native-safe-area-context` but does not wrap the root in a matching `SafeAreaView`, so no top-inset padding is applied globally.

**One-liner fix in `App.tsx`:** Replace the bare `<LauncherRoot />` with `<SafeAreaView style={{ flex:1 }} edges={['top']}>` from `react-native-safe-area-context`. That propagates the correct inset to every screen without touching any individual screen file.
