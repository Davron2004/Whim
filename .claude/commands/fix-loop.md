Run the parallel fix loop over the findings in "$ARGUMENTS" (a path to a findings list, or an inline list). You are the ORCHESTRATOR (main thread): you plan, dispatch, run the deterministic checks via `scripts/fixloop.sh`, adjudicate, and merge. You NEVER write fix code yourself, and you NEVER self-approve a protected-file change. Canonical design: docs/harness.md (full history: docs/archive/parallel-fix-loop.md — bare §refs below point there). Each run integrates on a staging branch, never on `main`: a STANDALONE batch cuts `integration/<batch-id>` from `main` at run start and exports `FIXLOOP_INTEGRATION_BRANCH=integration/<batch-id>`; a NESTED run (dispatched from a change's CLOSURE step in `.claude/commands/opsx/apply.md`) REUSES the change's active `integration/<change-id>` — never cut a second staging branch (one active at a time). `<run-branch>` below means that branch; `main` receives one human-ratified merge at run closure (docs/harness.md §3).

Preconditions (check once, STOP and tell the user if any fail):
- `.claude/settings.json` has `worktree.baseRef: "head"` (without it, fixer worktrees branch from a stale base).
- `scripts/gate.sh`, `scripts/gate-full.sh`, `scripts/fixloop.sh` exist and are committed clean (else the pinned-BASE tripwire refuses to run).
- STANDALONE run start: `git branch --list 'integration/*'` is empty (else a run is active — surface it); record `MAIN_TIP = git rev-parse main`, cut `git branch integration/<batch-id> "$MAIN_TIP"`, `git switch integration/<batch-id>`, ledger `run-start`. NESTED run: the change's active staging branch is already checked out — use it as `<run-branch>` and skip the cut.

You MAY run several findings in parallel — launch the fixers async and process each as its notification arrives — but step 8 (MERGE) is strictly serialized: one writer to `<run-branch>` at a time.

Batch bookkeeping (once, before any finding): the batch IS an OpenSpec change. `openspec new change fix-<batch> --schema whim-fixloop`, then fill the three artifacts in dependency order (`openspec status --change fix-<batch>` shows readiness): `findings.md` — the findings from "$ARGUMENTS", verbatim, mechanical-lane only; `plan.md` — the per-finding DONE specs from step 1 land here, under a tracked `- [ ] <id>` checklist (tick a box only at a TERMINAL ledger event, never on a worker's say-so); `dispositions.md` — initialize the ledger header BEFORE dispatching any worker, then append every disposition AS IT HAPPENS. The ledger + the fix/wip branches are the resume state if your context dies mid-batch. A fix that changes spec-visible behavior does NOT belong here — it needs a delta spec, i.e. a whim-harness change.

Per finding:

1. PLAN (read-only). Dispatch a read-only agent (`researcher` or `Plan`) to produce the DONE spec: the fix sketch, the **file allowlist** (glob patterns, one per line — save to a temp file), the **test** it must add, the expected red-without-fix, and **severity** (low/med/high). The orchestrator never explores the code itself.
   - **First, RECONCILE the finding against HEAD — findings lists go stale.** The planner must confirm the defect still exists at the current tip by reading the cited code (and, where the finding cites a fix-shaped change, `git merge-base --is-ancestor <fix-commit> <run-branch>`). If it's already fixed, the planner returns `ALREADY-FIXED: <evidence>` and you SKIP steps 2–8 for that finding (record it, do not dispatch a fixer). This is cheap here — no worktree is created — so always do it before the FIX step. **Then make it deterministic:** the planner's DONE spec must include a fenced EVIDENCE block — `## <repo-relative-path>` header lines followed by the verbatim buggy lines as they read at HEAD. YOU save the fence contents to a temp file and run `scripts/fixloop.sh stale <file>` before creating the worktree: exit 7 = evidence missing at HEAD → ledger `stale-skip`, do NOT dispatch; exit 0 = finding live. Prose judgment proposes; the stale check disposes.
   - **Protected-file scope (grants).** By default the allowlist contains NO protected files — a fix that would need one is a class-B PARK (mechanical fixes shouldn't fight the harness). The ONE exception: a **Class-1** config file (`package.json`/`package-lock.json`/`tsconfig*.json`/eslint/knip/`babel.config.js`/`metro.config.js`) may be declared in the allowlist IFF the edit *corrects* the config, not *relaxes a check* — apply the fix-vs-relax checklist (docs/harness.md) BEFORE declaring it. **Class-2** (gate scripts, `.claude/**`, `invariants/`, `build/`) is NEVER declarable — those are human-only, full stop. A declared Class-1 file triggers the GRANT step (1b).
   - **Classify the fix's testability and put it in the DONE spec** (the worker honors it — `.claude/agents/fix-worker.md` step 3). **Behavioral** (changes observable output / error surface / persisted state / executable control flow) → specify the red-without/green-with test (the `test` + `expected red-without` fields apply to this class only). **Structural, no behavioral delta** (rename, dead-code deletion, a dedup that can't change behavior) → specify **no test**; regression + inspection is the assurance — do NOT have the worker fabricate a source-grep. **Standing invariant** worth locking (single-source, no-`eval`) → specify a source/structural assertion that encodes the invariant, not the patch (litmus: would it read sensibly to someone who never saw the diff?); a new invariant/static-check file is owner territory, so don't ask a fixer for it. Rule of thumb: a source-string grep is legitimate ONLY as an invariant, never as a stand-in for an absent behavioral test. (Framework: docs/harness.md — test classification.)

1b. GRANT (conditional — skip unless step 1 declared a grantable Class-1 file). A grant is the ONLY thing that lets a fixer subagent edit a Class-1 config file; without it the edit is hook-blocked. Rules:
   - **Vet first (fix-vs-relax, docs/harness.md).** If the declared edit *weakens a checker* (eslint rule → off/downgrade, tsconfig strictness loosened, knip `ignore` added to silence a real finding, a script pulled from the gate, a dependency added) it is NOT grantable — PARK/escalate even though the plan named it. Only a *corrects-the-config* edit earns a grant.
   - **Write the grant** to `.claude/fixloop/grants/<wt-id>`, keyed to the worker's worktree id, containing ONLY the plan-declared Class-1 globs (one per line). This Write prompts YOU (main-thread `ask`) — that single approval, up front, is the human touchpoint; there are no per-edit prompts afterward. `.claude/fixloop/grants/**` is agent-unwritable, so the grant is unforgeable.
   - **Timing — prefer Flow A (pre-create).** The robust path: the orchestrator pre-creates the worktree (`git worktree add --detach .claude/worktrees/<id> <run-branch>`, id = finding id — needs sandbox off to check out `.claude/**`) and writes the grant BEFORE launching the worker into it; an orchestrator-owned worktree survives regardless of the worker's lifecycle. Flow B (let the worker class-B-stop on its first Class-1 edit — it reports its worktree root — then grant + `SendMessage` retry) is a fallback with a sharp edge: **an `isolation: worktree` tree is auto-removed when the agent's turn ends "if unchanged", so a worker that blocks on the config file before making ANY allowlisted change loses its worktree at the turn boundary and resumes rooted in the MAIN tree** (verified live 2026-07-01). Flow B is only safe once the worker's tree is dirty — so a worker that must class-B-stop on a grant need with an otherwise-pristine tree first writes an untracked `.gitkeep` in its worktree root to pin it (verified 2026-07-01: an untracked file defeats the non-`--force` auto-removal; it stays untracked, so it's invisible to the commit, the `integrity` diff, and the hermetic gate, and is discarded at teardown). In the semi-attended model (approve all plans + grants at setup, then leave), pre-create + grant every declared lane up front.
   - Class-2 is NEVER granted. A grant listing a Class-2 path is honored by neither hook nor `integrity` (it stays exit-3 tamper) — do not write one.

2. FIX. Record BASE = `git rev-parse <run-branch>` and append `dispatched` (finding, BASE) to the ledger, then launch a `fix-worker` subagent with `isolation: worktree` and the DONE spec. It comes up on a fresh worktree at `<run-branch>`'s tip (because of `baseRef: head` and the orchestrator keeping `<run-branch>` checked out), pins the tree with an untracked `.gitkeep` as its FIRST action (so a worker that dies before its first tracked change doesn't silently lose the worktree), builds, fixes, writes the test per its class, self-runs `./scripts/gate.sh` until green, commits, and reports. The completion notification gives you its `worktreeBranch` and `worktreePath` — append them to the ledger as `worktree-created`. (Planned upgrade, pending a human-applied bash-policy rule: orchestrator pre-creates every worktree — 1b's Flow A — which closes the die-before-first-commit hole structurally instead of by worker convention; see docs/archive/parallel-fix-loop.md §6.9.)
   - STATUS blocked (B/C) → adjudicate from the spec, or `SendMessage` the worker a corrected spec, or escalate to the user. Never improvise the fix yourself. **If the block is a plan-declared Class-1 file needing a grant** (worker reports "no grant covers this file" + its worktree root), do step 1b — vet, write the grant, `SendMessage` retry.

3. RED-CHECK (deterministic, the crown jewel) — **behavioral class only.** A `structural-no-test` fix has no test to red-check: SKIP this step and let VERIFY (inspection) + the FULL GATE staying green be the assurance. An `invariant` check asserts source, not behavior, so the reviewer judges it (don't expect a clean revert-RED). For a behavioral fix, from the report's TEST + PROD FILES:
   `scripts/fixloop.sh redcheck <worktreeBranch> <TEST...> -- <PROD FILES...>`
   exit 0 = RED (test fails without the fix → non-vacuous, good). exit 5 = GREEN = vacuous test → `SendMessage` the worker to write a real test (revision cap), else PARK.
   - **New-module caveat (the RED can be a false signal).** If the fix ADDS a new file that the test imports, reverting it to BASE deletes the module, so the test fails to *build* ("Could not resolve …") rather than *assert* — a coarse RED that proves nothing about the assertions. Re-run redcheck reverting ONLY the pre-existing prod file(s) so the suite still builds and the real assertion fails cleanly. A behavioral test exercising a brand-new helper can't be revert-red-checked at all (the helper vanishes) — note it and have the VERIFY reviewer judge that test's non-vacuity by inspection.

4. INTEGRITY (deterministic). `scripts/fixloop.sh integrity <worktreeBranch> <allowlist-file>` — **pass the PLAN's allowlist, never the grant.** They are two artifacts from one plan; passing the plan cross-checks grant ⊆ plan (a granted edit the plan didn't declare still trips tamper).
   - exit 0 = clean → eligible for orchestrator merge (subject to severity, step 7).
   - exit 6 = **sanctioned Class-1** — a plan-declared, granted config change ⊆ the allowlist. Legitimate, but it ALWAYS goes to the user at APPROVE (step 7); never self-merge a config change.
   - exit 3 = **tamper** — a Class-2 touch (even if it appears in the allowlist) or a Class-1 touch the allowlist didn't cover → **ESCALATE to the user** (never self-approve).
   - exit 4 = scope violation (a non-protected file outside the allowlist) → re-plan with a corrected allowlist if it stays same-subsystem & non-protected (§4.6), else escalate.

5. VERIFY. Dispatch the `reviewer` on `git diff <worktreeBranch>` against its merge-base (`git merge-base <worktreeBranch> <run-branch>`) + the DONE spec + the red-check result — NOT the worker's reasoning. Default-reject. Reject → `SendMessage` the worker the critique (revision cap = 2), then re-run from step 3. Cap hit → PARK.

6. FULL GATE (deterministic). `scripts/fixloop.sh gatefull <worktreeBranch>` (slower — Metro + Chromium). This runs `gate-full.sh` in a **fresh checkout of the committed branch tip**, NOT the worker's worktree: the worker's tree can hold untracked/gitignored files (poisoned `src/runtime/generated/*` or `build/` output) that the integrity diff can't see but the gate would execute against — so gating there could go green on code that isn't what you reviewed. The fresh checkout makes "what you verified" == "what you tested". FAIL → SendMessage the worker (one extended attempt), else PARK. (Do NOT fall back to `cd <worktreePath> && ./scripts/gate-full.sh` — that re-introduces the poisoning gap.)

7. APPROVE. low/med + everything green + integrity exit 0 → you may merge. **A sanctioned Class-1 change (integrity exit 6)**, high severity, or any escalation from step 4 → get the user's explicit OK first. A config change is ratified by a human at merge regardless of severity (§4.9) — this is the second and final human touchpoint after the grant approval.

8. MERGE (serialized — one finding at a time). `git switch <run-branch> && git merge --no-ff <worktreeBranch> -m "fix: <id>"`. **Then REGATE before the next merge:** run `./scripts/gate.sh` on the just-merged `<run-branch>` tip (fast, primary tree). Every fix was gated against its own BASE, never against the other fixes — the regate catches two individually-green fixes that break each other, at the merge that caused it instead of smeared across the batch end. PASS → ledger `merged` + `regate-pass`, then clean up: `git worktree remove --force <worktreePath>` and `git branch -d <worktreeBranch>`. FAIL → `git revert --no-edit -m 1 HEAD` (undo the merge), ledger `regate-fail`, park the branch. (`scripts/fixloop.sh finish <worktreeBranch> <allowlist-file>` re-checks integrity and prints these commands.)

On ANY terminal wall (verifier cap, unfixable-vacuous, scope/protected you won't pursue, persistent gate fail): `scripts/fixloop.sh park <branch> "<reason>"` — NEVER delete. The branch survives as `wip/<id>` (resumable) with a reason note under `.claude/fixloop/`.

Caps (bounded autonomy, then escalate — never silent-drop): the worker self-gates in its own loop; verifier revisions ≤ 2; one extended attempt on a gate cap-hit; then PARK. Protected-file touches and high-severity merges ALWAYS go to the user.

MEMORY (end of run, human-gated). Subagents never write the memory store directly — collect any `MEMORY:` proposals from their reports as you go. After the batch, dedupe them and apply the worthwhile ones yourself with the Write/Edit tool (each prompts you with the full diff — approve or skip). Follow the memory rules in CLAUDE.md (one fact per file + a one-line MEMORY.md index pointer). If running unattended (no human to approve), do NOT apply — list the proposals in the summary for the user to ratify. Never let a subagent's proposal land unreviewed.

The running progress note IS the change folder's `dispositions.md` — append every disposition AS IT HAPPENS (stale checks, worktree creation with BASE, check results, terminal events), never batched at the end. Tick plan.md checkboxes only at terminal events. At the end, give the user a summary: what merged, what parked and why, the memory proposals applied/pending, and what awaits their ratification — not the diffs — then archive the change (`openspec archive fix-<batch>`) once every finding is terminal.

CLOSURE (STANDALONE runs only — a NESTED run just returns control to the change's own CLOSURE step): after the last finding is terminal, run `scripts/fixloop.sh gatefull` on the run-branch tip if any merge landed, then follow the orchestrator-executed staging closure pipeline in `.claude/commands/opsx/apply.md` step 12 verbatim (ruleset probe → `git push origin <run-branch>` → `gh pr create --draft` → poll `gh pr checks` → on red: ingest with `scripts/sonar-pr-issues.mjs` → nested Sonar round → re-push → /git-cleanup with `target_branch=<run-branch>` → orchestrator force-push-with-lease → wait for re-analysis green → `merge-base --is-ancestor` check → `gh pr ready` + notify → the human's merge click on the PR → post-merge teardown). Automated on the ATTENDED host: the human's presence is required, their execution is not; the merge click on the reviewed PR is the sole ratification.
