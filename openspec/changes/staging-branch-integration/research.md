`fixloop.sh:38` `INTEGRATION_BRANCH="${FIXLOOP_INTEGRATION_BRANCH:-main}"` is already parameterized and used by `base_of()` (`:62`) and `finish`'s printed merge command (`:265`, uses `$INTEGRATION_BRANCH`). But `gatefull` (`:175-206`) hardcodes the concept of "the main tree" in prose/dies (`:181,186,188,196,199,202,204` — e.g. `die "main tree is dirty ..."`) though mechanically it just operates on "whatever tree is currently checked out" (doesn't literal-match a ref) — comments assume that tree is `main`.

Git-cleanup is fully `main`-hardcoded, not env-driven at all: `git-cleanup.md:1,3,20,28,29,48-49`; lane constants `:6-9` (`worktree: .claude/worktrees/main-squashed`, `branch: cleanup/main-squashed`); `git-cleaner.md:8,17,35,36` (hard rule: "Never any command naming `main`, `dev/v1` ... as a write target"); `git-cleanup-check.sh:16,22-29,48-50` pins `main_sha`/`main_tree`, resolves `refs/heads/main`, prints `git checkout main && git reset --hard cleanup/main-squashed` + `git push --force-with-lease origin main`.

`bash-policy.sh` tier-1 deny block `:124-130` literally lists `*"git branch -f main"*|*"git branch -D main"*|*"git branch -m main"*|*"git checkout -B main"*|*"git switch -C main"*` alongside `dev/v1` twins, plus a **blanket, ref-agnostic** `*"git push"*` (and fetch/clone/remote/etc.) deny that fires "for EVERY caller" regardless of target branch (`:115-122` comment, `:124`). `settings.json:44` mirrors this with `"Bash(git push:*)"` in `permissions.deny`. `invariants.yml:16-19` triggers CI `on.push.branches: [main]` + all `pull_request`.

`docs/harness.md` §3 (`:61-64`): "`main` is the integration branch — the trusted, single-writer target every fix/chain merges into... `fixloop.sh` reads `FIXLOOP_INTEGRATION_BRANCH`, default `main`." §9 (`:186-188`): future-upgrade note — "*Fully-unattended grant tier:* orchestrator auto-grants land on a throwaway dirty-area branch (set `FIXLOOP_INTEGRATION_BRANCH`), human reviews the whole branch before it touches `main`" — this is the closest prior design intent to the proposed change, but was never built.

## Constraints and invariants
- Pinned-BASE trust anchor (harness.md Principle 4, §3): BASE must remain a recorded SHA, recovered via `git merge-base <branch> <integration-ref>` — swapping the ref name is safe as long as `base_of()`'s `$INTEGRATION_BRANCH` stays the single source of truth; anything hardcoding `main` outside that var breaks the anchor.
- `git push` is denied to everyone today, unconditionally, by both `bash-policy.sh:124` (match-anywhere, no ref parsing) and `settings.json:44` — this is stated as a deliberate global backstop (harness.md §8: "`git push` is denied to everyone, so nothing reaches a shared remote without a human"). Any scoped allow of pushing `integration/*`/`chain/*` while keeping `main` denied is a **new capability class**, not a relaxation of an existing ref-aware rule — the current pattern has no ref-extraction machinery for `push` (contrast: the GIT_C worktree-path normalizer at `bash-policy.sh:20-38` is the only existing precedent for parsing structured args out of a git invocation).
- Class 2 files (`scripts/fixloop.sh`, `scripts/git-cleanup-check.sh`, `.claude/hooks/**`, `.claude/settings.json`) are never agent-editable (harness.md §4) — any change to these is human-ratified only, same as today.
- Git-cleanup's tip-tree-identity check (`git-cleanup-check.sh:26-39`) is pinned specifically to `refs/heads/main` and a hardcoded backup ref name — becoming branch-parameterized means threading a target ref through the grant file (`main_sha`→`target_sha`, plus resolving `refs/heads/main`→`refs/heads/$TARGET`), not just an env var, since the grant format itself names the fields.
- CI trigger (`invariants.yml:18`) only fires on push to `main` and on any `pull_request` — a staging branch push alone won't trigger `isolation-suite`/`quality-gate` unless either the trigger branches list gains the `integration/*` pattern or a PR is opened against it (which the SonarCloud flow implies anyway).

## Integration points
- `scripts/fixloop.sh:38` (`INTEGRATION_BRANCH` var) is the existing parameterization seam; `base_of()` (`:62`), `finish` (`:265`) already consume it. `gatefull`'s hardcoded prose/dies (`:181-204`) don't reference the var by name but describe "the main tree" — would need re-phrasing/generalizing since the mechanism just uses whatever tree is checked out currently, not literally `main`.
- `apply.md` and `fix-loop.md` have no variable indirection at all — every `main` occurrence listed above is inline prose/commands; introducing a per-run `integration/<change-id>` branch means rewriting each cited line to reference a computed variable, and adding the orchestrator step that cuts `integration/<change-id>` from `main` at run start.
- `bash-policy.sh:124-130` tier-1 deny patterns for branch force-ops explicitly enumerate `main`/`dev/v1` by name; a parallel pattern set (or a prefix-glob) would be needed to protect `integration/*` similarly once it's user-facing, while a *narrower* new allow-branch (before the blanket push/branch denies) would be needed to permit push/branch-mutation on `integration/*`/`chain/*` refs specifically — same file, same case-statement structure, sits above the existing tier-1 block.
- `settings.json:43-46` `permissions.deny` blanket `Bash(git push:*)` is the second enforcement point mirroring bash-policy; a scoped allow would need a corresponding `permissions.allow` entry with a ref-glob Claude Code's Bash matcher can actually express (verify: Claude Code's `Bash(cmd:*)` matcher is a prefix match on the whole command string, not ref-aware — I did not verify whether it supports finer-grained argument matching).
- `invariants.yml:18` trigger block is the sole integration point for making CI fire on `integration/<change-id>` pushes (either widen `branches:` to a glob or rely purely on the draft-PR `pull_request` trigger already present).
- Git-cleanup lane (`git-cleanup.md`, `git-cleaner.md`, `git-cleanup-check.sh`) is the most structurally distinct integration point: its grant file schema (`main_sha`/`main_tree`), hardcoded lane names (`main-squashed`), and hard rule text naming `main` explicitly would all need a target-branch parameter threaded through, not just an env default.
- SonarCloud is referenced only in `docs/harness.md` (§4 row, "SonarCloud (external)") among harness docs proper — it is server-side, PR-triggered, has no local script/config (`sonar-project.properties` deliberately absent per that row) and no other harness file wires it up; the draft-PR requirement in the proposal is new plumbing, not an existing hook to redirect. `gh` (GitHub CLI) appears nowhere in `.claude/settings.json` or `bash-policy.sh` except the credential deny-list entry `settings.json:17` (`GH_TOKEN` env var denied to sandboxed processes) — no existing `gh`-invocation allowlisting exists for opening/managing a draft PR.
- **Smallest-change surface:** MUST change — `scripts/fixloop.sh` (generalize `gatefull`'s "main tree" assumption + wire cleanup-lane-style parameterization), `.claude/commands/opsx/apply.md`, `.claude/commands/fix-loop.md` (every literal `main` cited above), `.claude/hooks/bash-policy.sh` + `.claude/settings.json` (new scoped push/branch-op allow for `integration/*`/`chain/*`, main still denied), `.claude/commands/git-cleanup.md` + `.claude/agents/git-cleaner.md` + `scripts/git-cleanup-check.sh` (branch-parameterize the grant/lane), `.github/workflows/invariants.yml` (trigger branches). MERELY-mentions (update prose only, no mechanism change) — `docs/harness.md` §3/§9, `CLAUDE.md`'s harness summary, `.agents/skills/source-command-opsx-apply/SKILL.md` (regenerates from schema, don't hand-edit).

## Risks and unknowns
- I did not verify whether Claude Code's Bash permission matcher (`Bash(git push:*)`) can express ref-scoped patterns (e.g. allow `git push origin integration/*` while denying `git push origin main`) — this may force the ref-parsing logic into `bash-policy.sh`'s existing `POLICY_CMD` normalization instead of `settings.json`.
- I did not verify what, if anything, in `openspec/schemas/whim-harness/schema.yaml` or `whim-fixloop/schema.yaml` encodes `main` as an assumption (e.g. `apply.instruction` routing) — the digest only covered the two runbooks explicitly named in the ask.
- I did not trace how a draft-PR-based SonarCloud iteration step would interact with the existing `gh`-absent tooling — no script in the repo currently opens/updates a PR; this is greenfield, not a hardcoded assumption to unwind.
- `gatefull`'s "dirty main tree" check (`fixloop.sh:187-188`) operates on whatever is the *current* checkout, not a named ref — I did not fully verify there's no other spot where it implicitly assumes the checked-out branch is literally named `main` (e.g. via `git symbolic-ref`) beyond the trap/restore logic already read.
