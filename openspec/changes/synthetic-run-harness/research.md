## Current behavior
`build/build.mjs` esbuilds the runtime parts (neutralize/resolver/probes/syscall/loader) and every fixture app into IIFEs (classic JSX, externals `vc-sdk`/`react`/`react-dom`, `tsconfigRaw:'{}'`), writing `src/runtime/generated/runtime-artifacts.json` as `{parts, bundles, appRecords}` — this is the one artifact both invariant suites and (per roadmap #10) the synthetic-run harness read to assemble scenario pages "against THIS runtime," never a re-derivation.

`assemble.mjs` exposes two pure functions: `buildSrcdoc({parts, channel, bakedBundle?})` returns the iframe document (locked CSP + ordered inline scripts); `buildOuterHtml({srcdoc, bundles, initial, channel, showDiagnostics=true, autostart=true, syscallSink='rn'})` returns the WebView-page HTML with the nonce handshake, on-screen diagnostics, and `window.__whimControl` (`reinject`, `deliver`, `navBack`, `listBundles`, `setGeneration`). `syscallSink:'exposed'` routes bundle syscalls to a Playwright-`exposeFunction`-bound `window.whimHostDispatch` instead of the RN relay — this is the seam the bridge-invariants suite already uses and the synthetic-run harness would reuse for its real bridge+`:memory:` storage.

`run-against-build.mjs` shows the exact boot recipe: `chromium.launch()` → `browser.newPage()` → attach `page.on('console'...)`/`page.on('pageerror'...)` → `page.goto(pathToFileURL(file).href, {waitUntil:'load', timeout:20000})` → `page.waitForFunction(() => title !== 'WHIM:pending' || /refus|REJECTED|BLOB/i.test(delivery), {timeout:12000}).catch(()=>{})` (the silent fall-through: a hung page is NOT itself a failure signal — subsequent DOM reads just see stale/absent values, so the invariant suite's own assertions are what fail, not the wait itself) → optional `drive(page)` → fixed `waitForTimeout(500)` → read DOM (`#status/#probes/#delivery/#paint`) + iterate `page.frames()` for iframe text. Chromium comes from `playwright ^1.49.0` (devDependency, already present at root — no new dep needed for headless-Chromium boot).

`loader.js`'s authenticated vocabulary (all wrapped `{__whimHarness:true, nonce, kind, payload}` except the pre-auth `hello`): `hello` (no nonce, first frame, triggers host's `__whimHostInit` reply), `ready` (posted after host-init lands, triggers pending delivery), `delivery` (`{accepted, generation, note?}` or blob-refusal shape), `paint` (`{generation, mountToFirstPaintMs, appName}`), `error` (`{where, name?, message}` — this is where mount/deliver/probes failures surface, not a separate warning channel), `probes` (`{contained, passed, total, failures[], t7?, ...}` from the closure-captured trusted probe fn). Capability-bridge syscalls (`m.whim==='syscall'`) and nav-depth hints (`m.__whimNavDepth===true`) ride an unauthenticated, `ev.source`-verified side channel in the outer page's message listener (not the `__whimHarness` nonce path) — gate *denials* themselves don't have a dedicated frame kind; a denial comes back as a `sysret` over the syscall relay, observable only by whoever owns the dispatch side (RN or the exposed Node host).

`server/src/pipeline.ts`'s `Pipeline.run(request, signal?): AsyncIterable<GenerationEvent>` stub emits `stage:{stage:'run', status:'start'|'done'}` with no actual sandboxed execution behind it — this is precisely the stage roadmap #10 will eventually feed. `contract/src/index.ts`'s `GenerationEvent` is a zod `discriminatedUnion` on `type` with variants `stage|token|diagnostic|usage|result|failure`; `Diagnostic = {kind:string, severity?, message?, symbol?, line?, hint (required, min 1)}` — `kind` stays an **open wire string** (contract's own comment: "#9 re-exports the static-check closed kind union as a TS-only narrowing").

`checks/contract.ts` owns the actually-closed `DIAGNOSTIC_KINDS` array (`typeof DIAGNOSTIC_KINDS[number]` derives `DiagnosticKind`), split into "verbatim-reused" (matches runtime/engine kind strings, e.g. `undeclared_capability`) and "new" static-only kinds. `harness-diagnostics/spec.md` requirement 2 explicitly states: "Downstream stages (synthetic run, repair loop) SHALL extend the vocabulary additively through this module, never by minting ad-hoc kind strings elsewhere."

The bridge Node-acceptance pattern (`src/host/bridge/test/acceptance.ts`) and the bridge-invariants suite both show the full recipe for a real, non-mocked bridge+storage stack under Node: `createEngine(createNodeSqlExecutor(':memory:'))` as an `EngineFactory`, `launchApp(app, memFactory)` → `{ok, realm}`, `Dispatcher.forRealm(realm, registry, permissionHook?)`, then either call `dispatcher.handle(frame)` directly (Node acceptance) or expose `host.dispatch` to a Chromium page via `page.exposeFunction` (bridge invariants, when the bundle is delivered live in a WebView-driven iframe).

## Constraints and invariants
- Never widen the CSP; never value-replace `Function`/`eval` (docs/decisions #35/#37; assemble.mjs `LOCKED_CSP` comment) — the harness must consume `buildSrcdoc`/`buildOuterHtml` unmodified, never fork a "looser" page for testability. Roadmap #10 states this explicitly: "never loosen the page/CSP to ease testing."
- Trusted vantage only (F4, spike2 constraint #3): observation must come from the closure-captured probe/paint/error frames the outer page already authenticates by nonce, never the bundle's self-report. `contained`/`probes`/`paint`/`error` are the only nonce-authenticated frames; syscall/nav-depth frames are `ev.source`-verified but NOT nonce-authenticated (a hint, not authority — F4 discipline still applies: the harness must not treat them as ground truth for anything beyond routing).
- Realm reset = recreate the iframe, never re-inject in place (spike2 constraint #5 / T7) — a deterministic interaction sweep across screens that needs a fresh generation must go through `__whimControl.reinject({reset:true, ...})`, not same-realm re-delivery.
- Five spike2 carry-forward constraints (titles only): (1) window-level value-strip, never lexical shadow; (2) loader/SDK holds no capability stronger than `parent.postMessage`; (3) trusted-vantage verdict (F4); (4) authenticate/ignore bundle-origin control messages by nonce, not by `kind` tag (T6); (5) re-injection resets the realm (T7).
- `invariants/` and `build/*` are agent-protected paths (`.claude/hooks/protect-harness.sh`) — the harness must be new code that *imports/consumes* `build/assemble.mjs` and the invariants' patterns, never edits them (docs/harness.md).
- Decision #42 / roadmap #10: this is explicitly server-side-Chromium tier; on-device stays acceptance-only (mirrors the invariants-suite framing: "Desktop is the FAST FILTER... the authoritative pass is on-device").
- `Diagnostic.kind` is open on the wire (`contract/src/index.ts`) but closed at the static-checks layer (`checks/contract.ts` `DIAGNOSTIC_KINDS`); harness-diagnostics spec requirement 2 mandates new runtime-observed kinds (throws, unhandled rejections, gate denials, containment failure) be added additively to that same module, reusing runtime kind names where one already exists (e.g. bridge's `undeclared_capability`, `permission_denied`, `invalid_params`, `unknown_method`, `handler_error`, `malformed_envelope` from `gate.ts`/`dispatcher.ts`).
- `checks/` is a plain top-level directory (not an npm workspace) by design (#9's contract notes) — deliberately dependency-light so non-workspace consumers can import it as TS source.

## Integration points
- Page/runtime boot: `buildSrcdoc`/`buildOuterHtml` from `build/assemble.mjs`, fed `src/runtime/generated/runtime-artifacts.json`'s `{parts, bundles}` (or a single new candidate bundle string standing in for one `bundles[name]` entry) — same shape as `run-against-build.mjs`'s `pages` construction.
- Bridge/storage wiring: `launchApp(appRecord, () => createEngine(createNodeSqlExecutor(':memory:')))` → `Dispatcher.forRealm(realm, createDefaultRegistry(), permissionHook?)`, exposed to the page as `whimHostDispatch` via `page.exposeFunction`, with `buildOuterHtml({..., syscallSink:'exposed'})` — the bridge-invariants `host-shim.ts`/`runner.mjs` pair is the load-bearing precedent.
- Diagnostics emission: new runtime-observed kinds added to `checks/contract.ts`'s `DIAGNOSTIC_KINDS`, surfaced through `Diagnostic` (`checks/contract.ts` shape: `{kind, severity, line, column?, symbol?, message, hint}` — note this static shape requires `line`/`severity`, unlike the looser wire `Diagnostic`; a runtime-observed diagnostic without a source line needs to fit the wire shape, not the static one, or the two shapes need reconciling).
- Server-side hookup: `server/src/pipeline.ts`'s stubbed `run` stage is the eventual caller (roadmap #11, out of scope for #10) — #10 is explicitly "lib-first," no hard dependency, but its output feeds that stage's replacement.
- Workspace placement: root `package.json` `workspaces: ["contract", "server"]`; `checks/` is a non-workspace plain directory precedent for "importable dependency-light TS source" that doesn't need a workspace entry. `playwright` is already a root devDependency (used by `invariants/`), so a server-adjacent Chromium-driving lib can reuse it without adding it to a new workspace — but if the harness becomes its own npm workspace, `guard:metro` (docs/harness.md §4 — the CI byte-size assertion that `contract`/`server` don't change what Metro resolves for the RN bundle) would need to hold for a new workspace too.
- Nav sweep: the "visit every screen" sweep depends on `sdk-navigation` landing first — today `loader.js`'s nav-depth seam is a stub anchor only (comment: "no SDK nav exists yet... depth is always 0"), and `sdk-navigation`'s change folder contains only a bare `.openspec.yaml` scaffold (schema `whim-harness`, no design/tasks yet).

## Risks and unknowns
- I did not verify the exact wire shape `NavDepthFrame` will take once `sdk-navigation` lands — the current loader.js comment describes intent (`{__whimNavDepth:true, depth, generation}` posted on every depth change) but this is pre-implementation; the harness's "visit every screen" sweep logic depends on whatever `sdk-navigation`'s design.md ultimately specifies.
- I did not verify whether gate *denials* get a dedicated outer-page frame kind today — currently a denial is just the `sysret` returned from `dispatcher.handle`/`whimHostDispatch`, observable only to whoever owns the syscall-sink side (there is no `kind:'gate-denied'` frame in loader.js's vocabulary); the harness will likely need to read denials directly off its own exposed dispatch function rather than off an authenticated frame.
- I did not verify how "enumerate every interactive SDK element" would introspect the DOM/SDK component tree — no existing suite does a generic enumerate-and-tap sweep; `run-against-build.mjs`'s `drive` callbacks manually target a single `button` per fixture. Whether the SDK exposes stable selectors/roles for this is unconfirmed.
- The static (`checks/contract.ts`) vs wire (`contract/src/index.ts`) `Diagnostic` shapes differ (required `line`/`severity` vs optional) — I did not verify which one a runtime-observed synthetic-run diagnostic is meant to conform to; this looks like an open reconciliation point rather than a settled fact.

## Open questions for the planner
- Should runtime-observed diagnostic kinds (throw/unhandled-rejection/containment-failure/etc.) live in `checks/contract.ts`'s `DIAGNOSTIC_KINDS` (as harness-diagnostics req 2 literally mandates) even though that module currently only describes *static*-check kinds and has a stricter required-`line` shape than the wire `Diagnostic`?
- Does this change wait for `sdk-navigation`'s design to land (even just its contract, per the assumption stated in the task), or can it stub the nav-sweep against today's always-depth-0 seam and extend later?
- Is the harness itself a new npm workspace (subject to `guard:metro`) or a plain top-level directory like `checks/` (dependency-light, no workspace entry)?
