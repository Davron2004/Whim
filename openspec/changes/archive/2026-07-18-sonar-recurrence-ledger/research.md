## Current behavior
- **Critic** (`.claude/agents/critic.md:22-38`): writes one report per run to `openspec/critic/<YYYY-MM-DD>.md` with sections `# Critic report <date>` / `Scope: <ref range>, <n> commits, <m> files` / `## Findings` (six numbered categories, each entry `### [severity: high|med|low] <title>` with `Where/Category/What/Why it matters/Suggested approach`) / `## Patterns worth a tripwire` / `## Not findings` (≤3). Cap: 15 findings, ordered by severity, overflow noted as a count. Critic is read-only (Read/Grep/Glob/Bash only) and "never fixes anything."
- **critic-run.md** (1 line): dispatches critic over everything since the newest file in `openspec/critic/` (or last archive tag, or asks); gives it the ref + `openspec/critic/<today>.md` as the target path; summarizes finding counts/severities/tripwire patterns back to the user — does not act on findings itself.
- **openspec/critic/README.md**: confirms one file per run, `<YYYY-MM-DD>.md`, "since the last report" = newest file present, git-tracked deliberately so the marker persists.
- **fix-loop findings input** (`fix-loop.md:9`): a findings list — e.g. `openspec/changes/archive/2026-07-12-fix-sonarjs-gate/findings.md` — is free-form markdown, `## F<n> — <path>:<line> — <rule-id>` header then prose (fix intent + "Behavioral net: <test cmd>"); no rigid schema, just per-finding heading + file:line + rule.
- **`fixloop.sh stale`** (lines 209-237): reads an evidence file shaped `## <repo-relative-path>` header lines followed by verbatim trimmed source lines; checks each line still exists in `git show HEAD:<file>` via fixed-string grep; exit 0 = still present (live), exit 7 = missing (likely already fixed, don't dispatch). It does NOT parse rule IDs, dates, or run IDs — it only cares whether cited source text still exists at HEAD.
- **dispositions.md ledger** (`openspec/schemas/whim-fixloop/templates/dispositions.md`): append-only, one line per event, format `- <UTC time> <finding-id> <event> — <detail>`, fixed event vocabulary (`stale-ok|stale-skip|worktree-created|fix-reported|redcheck-red|...|merged|parked|escalated|skipped`), exactly one terminal event per finding. This is per-batch (lives in the change folder), not cross-run/cross-change.
- **`.eslintrc.js`** (32 lines): `extends: ['@react-native', 'plugin:sonarjs/recommended-legacy']`, one custom `no-restricted-syntax` rule (bare `.sort()`), overrides for `.mjs`/`scripts`/`build` (node env) and `checks/test/**` (`sonarjs/assertions-in-tests: off`).
- **`scripts/gate.sh:56`**: `check "lint" npm run -s lint -- --max-warnings 0` — the one lint invocation, zero-warning enforced.
- **SonarCloud row** (`docs/harness.md:99`, verbatim): "SonarCloud (external) | GitHub PR quality gate | Automatic analysis on every push to a branch with an open PR — server-side, no repo config (`sonar-project.properties` deliberately absent), not runnable locally or in the deny-egress container. ## Constraints and invariants
- Critic is categorically read-only and "never fixes anything" (`.claude/agents/critic.md:8`) — a recurrence ledger the critic reads must not require it to write anywhere except its own dated report.
- `.eslintrc.js` is Class-1 (`scripts/fixloop.sh:56` `CLASS1` array includes `'.eslintrc*'`; also `docs/harness.md` §4/§8) — grantable only via a fix-vs-relax-vetted, human-ratified edit, never a plain agent write. Any lint-promotion (rule ID → local eslint rule) is therefore a Class-1-touching change, never mechanical-fix-loop material by default.
- `.claude/fixloop/**`, `scripts/gate*.sh`, `scripts/fixloop.sh` itself are Class-2 (`scripts/fixloop.sh:47-53`, `docs/harness.md` §4) — never agent-editable regardless of what the ledger says.
- `fixloop.sh stale`'s evidence-file grammar (`## <path>` header + verbatim lines) is a hard parsing contract already depended on by the fix-loop; a recurrence ledger is a separate artifact and must not silently reuse/collide with this format unless intentionally made stale-check-compatible.
- `openspec/critic/` is git-tracked specifically so "newest file" scoping works (README.md) — any ledger placed there must not break the "newest `<date>.md` = last report" heuristic `critic-run.md` uses.
- SonarCloud itself is external, server-side, not locally runnable (docs/harness.md:99) — any ledger-population step that wants live SonarCloud data (rule/finding IDs) cannot run inside the gate or the deny-egress container; today that data enters the repo only via human-transcribed prose in change folders (see `clear-sonarqube-warnings/research.md`'s finding ID `AZ9ciqsaZpP-TVNR4hrv`).
- Dispositions ledgers today are scoped per fix-loop batch (inside `openspec/changes/<batch>/dispositions.md`), not cross-batch/cross-time — a recurrence ledger spanning "every SonarCloud fix round" is a new kind of artifact, not a reuse of the existing per-batch ledger.

## Integration points
- **Critic side**: a new report section would attach in `.claude/agents/critic.md` after the existing six-category findings block (currently ends `## Patterns worth a tripwire` / `## Not findings`, lines 33-38) — a "recurring external findings" section is structurally a peer of those, and `critic-run.md`'s one-line orchestration instruction would need updating if the critic must additionally read the ledger before writing its report.
- **Ledger population side**: nothing today writes any cross-run Sonar record. The only existing artifacts that mention a specific SonarCloud rule/finding ID at all are prose inside `openspec/changes/*/research.md` and `openspec/changes/archive/2026-07-12-fix-sonarjs-gate/findings.md` — both per-change, not appended anywhere durable. A ledger-append step would be new: either a fix-loop step (`fix-loop.md`, after step 8 MERGE or as part of dispositions bookkeeping) or a standalone script, since `scripts/fixloop.sh` is Class-2 and can't be extended by a feature-implementing agent without human ratification.
- **Location convention**: `openspec/critic/` is the closest existing "durable, git-tracked, dated" convention (README.md explicitly says it's kept in git so the marker persists); a ledger could live there as a non-dated file (e.g. sibling to the dated reports) so the critic can read it without crossing subsystem boundaries, OR under `docs/` alongside `decisions.md`/`DEVLOG.md` if it's meant to read as project history rather than harness state. Both are plausible given conventions; nothing in the repo picks one today.
- **Which files MUST change** for this proposal's mechanics to exist: (1) wherever the ledger file itself lives (new file, not yet decided by convention — see above); (2) `.claude/agents/critic.md` to add the "recurring external findings" section and its ≥N-runs promotion-candidate logic; (3) `.claude/commands/critic-run.md` if the orchestration step must now also pass/read the ledger path; (4) whichever fix-loop or Sonar-fix-round artifact is designated as the ledger-appender (likely a `fix-loop.md` step addition, since that's the existing per-round mechanism for Sonar-driven findings) — note `.claude/commands/fix-loop.md` IS listed as Class 2 in `docs/harness.md` §4 — "`.claude/**` (hooks, settings, agents, commands, grants)" — so editing it is human-ratified, not agent-mechanical, same as critic.md.
- Today, findings enter change folders by a human transcribing the SonarCloud PR UI (per `clear-sonarqube-warnings/research.md`'s explicit "I did not verify a fresh SonarCloud scan... GitHub API access was unavailable" note) — there is no automated Sonar→repo pipe. After this change, the same human-transcription step would presumably also append to the ledger; no code path automates the append.
