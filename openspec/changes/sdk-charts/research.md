## 1. Corpus demand
`docs/app-corpus.md` pinned notes (quoted): *"Chart is its own OpenSpec change (`sdk-charts`): the corpus needs exactly three shapes — bar, line, calendar-heatmap. Declarative, data-as-props, tokens-only. No pie/scatter, no tooltips/pan/zoom, no animation, no canvas in v1."* Two Tier-0 corpus rows pull it: spending tracker ("**Chart(bar,line)**") and habit tracker ("**Chart(heatmap)**, a second chart shape").

`docs/sdk-gap.md` §4 assignment (quoted): *"→ #4 `sdk-charts`: one `Chart` export, `kind: 'bar' | 'line' | 'heatmap'` (single doc entry is the model-friendly shape; final call in #4's design). Exactly these three kinds."* §5 budget: the v1 export ceiling is 42 (ceiling was 60–80), computed as *"29 components (the 7 existing + 20 from §4 + `Chart` + `Grid`/`Spacer` counted above) + `defineApp` + `storage` + 5 hooks + 2 nav + 2 effects + 2 cues = 42"* — Chart is a pre-reserved single export in that budget. `sdk-design-system`'s own design.md reports it landed "≈35" exports, so Chart (1 export) lands comfortably under ceiling. No sparkline/progress-ring/donut anywhere — `ProgressBar` already covers the linear-progress case; nothing else asks for a ring/donut.

## 2. Current SDK surface & conventions
`index.tsx` exports: `useState/useEffect/useRef`, `delay`, `interval`, `defineApp`, `storage` (kv+records facade), `cues` (haptic+sound), `Screen/Stack/Row/Text/Heading/NumberInput/Button`, re-exported from `controls.tsx` (`TextInput/Switch/Checkbox/Slider/SegmentedControl`) and `surfaces.tsx` (`Card/Divider/Spacer/Grid/Badge/ProgressBar/List/ListItem/EmptyState/Modal`). Every component takes **tokens, not values** and resolves them via `tokens.ts` functions (`space()/radius()/color()/weight()/textSize()`), which read the active `WhimTheme` (theme.ts). Size/tone/variant props are string-literal unions with sensible defaults (e.g. `Button.variant: 'primary'|'secondary'|'ghost'|'danger'`, `Badge.tone`, `ProgressBar.tone`). Rendering is always `React.createElement(tag, {style:{...inline...}}, children)` — no CSS classes, no stylesheets, every style object built per-render from token resolvers. `events.ts` gives `emitUiEvent('press', label)` (one-way telemetry over the same transport); `press.ts` gives `usePressed()` + `TAP_RESET` for the opacity-dip tap affordance used by Button/Card/ListItem.

`WhimTheme.colors` is a **closed** 10-key set: `text, text-muted, primary, on-primary, bg, surface, border, danger, positive, warning`. There is **no per-series/chart-specific color role** — a multi-bar chart or multi-line chart has no obvious "series 2/3" token today.

## 3. Style gallery fixture & seeding
`style-gallery.app.tsx` is one file, `capabilities: []`, single `Home` screen, sections wrapped in `Card`+`Stack`, and all demo data is local `useState` (numbers/strings) — **not** storage-backed. "Gallery seeding" (`seed.ts`, `SEED_VERSION`) is host-level app-installation (installs the bundle as a launchable example via `StoreAccess.install`), **not** data-record seeding — it has nothing to do with populating a mini-app's own storage collections. There is no host-level mechanism for pre-populating a fixture's storage at first run; the only existing pattern (`water-counter.app.tsx`) is app-level self-seeding: declare `capabilities:['storage']`+`schema`, then in a mount `useEffect` load and, if empty, `storage.records.append(...)` itself.

## 4. Data path for charts
`storage.records.list(collection, query?): Promise<StorageRecord[]>` where `StorageRecord = {id:number} & {[displayField:string]: JsonValue}`, `ListQuery = {where?, orderBy?, limit?, offset?}` (equality/gt/gte/lt/lte, AND-only, no `contains`). Every existing fixture (water-counter, latency-probe) follows the same idiom: `list()` in a `useEffect` → `setState` with a plain derived array/number → pass to a component as an ordinary prop. **No** SDK component or hook binds directly to storage today. This confirms Chart's data prop is expected to be a plain array the app author builds itself from `StorageRecord[]` (e.g. `{label,value}[]` or `{date,value}[]`) — there is no other data-binding mechanism to displace.

## 5. Spec home
`sdk-design-system` is **not yet archived** — it lives at `openspec/changes/sdk-design-system/` (not `openspec/changes/archive/`), and `docs/capabilities.md` does not yet list it. It created a **new** capability spec, `sdk-design-system` (`.../specs/sdk-design-system/spec.md`), rather than extending `sandbox-rendering`. `openspec/specs/sandbox-rendering/spec.md` remains the original v0.1 spec (React-to-DOM path, event round-trip, first-paint ceiling, NumberInput focus fix) and holds no per-component-kit requirements. Precedent set: component-kit-shaped requirements go in the dedicated `sdk-design-system` capability, not `sandbox-rendering`.

## 6. Rendering constraints
`LOCKED_CSP` (build/assemble.mjs, decision #35, never widen): `default-src 'none'; script-src 'unsafe-inline'; style-src 'unsafe-inline'; img-src data:; connect-src 'none'; base-uri 'none'; form-action 'none'; frame-src 'none'; child-src 'none'; worker-src 'none'`. No `unsafe-eval`/`blob:`/`data:` in `script-src` ever. No remote fonts (`FONT` = `system-ui` stack), no remote images (`img-src` is `data:` only), no network (`connect-src 'none'`) — charts must be entirely self-contained DOM/inline-SVG with inline styles, no external assets. `docs/app-corpus.md` additionally rules out `<canvas>` as a product decision (not a CSP restriction — canvas itself isn't CSP-blocked). Decision #44's corpus-need rule (quoted): *"each addition needs (a) the no-ambient-authority test ... and (b) a concrete Tier-0 corpus need, recorded here."* Chart already satisfies (b) via `docs/sdk-gap.md`/`docs/app-corpus.md`; it trivially satisfies (a) since it's pure rendering. `sdk-design-system`'s off-limits list (CSP, sandbox attrs, resolver allowlist, bridge contract/registry/gate/dispatcher, version store, storage engine, `scripts/gate*.sh`, package.json, invariant suites) carries forward as the standing boundary for any SDK-only change.

## 7. SDK reference doc
`docs/sdk-reference.md` is **hand-maintained, not build-generated** (explicit header: "Mirrors src/sdk/{index,controls,surfaces,tokens,theme}.ts verbatim — never invents a prop/default"). Structure: §1 mini-app contract + minimal example (points to style-gallery as canonical), §2 Components as markdown tables (`Component | Prop | Type | Default | Semantics`, grouped Core/Controls/Surfaces), §3 Tokens (5 scale tables), §4 Hooks & effects (table), §5 Capability facades (verbatim TS signatures in code fences), §6 Theming (prose). A Charts section fits as a new §2 subsection; its data-prop shape (array-of-points) likely needs a code fence rather than a table row, mirroring how §5 breaks from the table format for multi-line signatures.

## 8. Testing conventions
sdk-design-system's only automated test is `src/host/launcher/test/theme.suite.ts` — a Node suite (Harness `ok/eq/test/throws`) covering pure theme-model logic (`resolveTheme`/`sanitizeTheme`/`shellPalette`/pref persistence), run via `npm run launcher:test`. **No per-component DOM-rendering unit tests exist anywhere** — visual/behavioral correctness of Button/Card/List/etc. rides the style-gallery fixture (manual QA + knip anchor) plus `npm run build` (typecheck/esbuild/source-map) and the unmodified Chromium `npm run invariants` (proves containment, not visual correctness). There is no `src/sdk/test/` directory today. For sdk-charts: any pure data-to-geometry math (bucketing, scaling, heatmap-grid layout) is a Node-suite candidate (new file, following the `theme.suite.ts` pattern); visual/containment correctness would again ride `npm run build` + `npm run invariants` + a gallery-style demo fixture — no new Chromium suite is implied since Chart adds no containment-relevant surface. Exporting `Chart` from `index.tsx` (or a sibling module re-exported through it, matching `controls.tsx`/`surfaces.tsx`) is sufficient for the build; no `resolver.js` allowlist change is needed (Chart would only import `react`, already allowlisted). `build/build.mjs`'s hand-written `APPS`/`bundles` maps (lines ~188–202/283–291, **hook-protected, main-thread-only**) only need touching if a new fixture file is added.

## 9. Manifest/capability angle
Chart is pure rendering — no bridge/syscall involvement, matching every existing display-only component (Card/List/Badge/ProgressBar touch no capability). No registry/gate/dispatcher change. Build-time manifest extraction (`build/build.mjs`) only reads each fixture's own `defineApp({capabilities, schema})` declaration (re-bundled per-fixture with react/react-dom stubbed) — it is indifferent to which SDK components a fixture imports. A Chart-only demo fixture with `capabilities: []` extracts identically to style-gallery/tip-splitter; a storage-backed chart demo would need `capabilities:['storage']`+`schema` exactly like water-counter, with no chart-specific wrinkle.

## Open questions for the planner
1. No "series 2/3" color token exists (10 fixed semantic roles only) — does Chart reuse existing roles (e.g. `primary` for a single series, since the corpus doesn't call for multi-series) or does this change need a new token/role?
2. `sdk-design-system` isn't archived yet — should sdk-charts' spec delta target the not-yet-archived `sdk-design-system` capability directly, or wait for archival / mint its own capability the way sdk-design-system did relative to `sandbox-rendering`?
3. Are heatmap cells interactive (habit tracker's "tap a day" isn't specified in the gap doc, which only excludes tooltips/pan/zoom) — confirm v1 heatmap cells carry no `onPress`.
4. No host-level mechanism exists to auto-seed a fixture's storage; if a Chart demo fixture wants realistic data it must self-seed exactly like water-counter — confirm the demo fixture uses hardcoded arrays (simpler, `capabilities: []`) rather than a storage-backed demo.
5. `useMemo`/`useCallback`/`useReducer` and navigation were scoped to sdk-design-system by `docs/sdk-gap.md` but did not ship (absent from `index.tsx`, silently missing from design.md's Non-Goals) — if Chart needs derived-value memoization internally, does that require its own #44 corpus-need justification, or can Chart just compute inline without a new hook export?
