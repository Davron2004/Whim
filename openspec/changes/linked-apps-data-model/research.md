# Research digest: linked-apps-data-model — a fork/rewind-created app sharing the original's storage-engine database

## Relevant files
- `src/host/launcher/store-access.ts` — `StoreAccess.engineAppId`, `fork`, `remove`; the only sanctioned launcher→version-store path.
- `src/host/launcher/app-index.ts` — `InstalledApp` shape, `AppIndex.refCount`.
- `src/host/launcher/useMiniAppHost.ts` — where `engineAppId` is threaded into `launchApp`/`createStorageEngine` at realm-bind time.
- `src/host/bridge/launch.ts` — `launchApp`: engine creation + `schemaArtifact` open sequencing, per-realm binding.
- `src/host/bridge/contract.ts` — `AppRecord`/`RealmRecord` shapes (`appId` field, no per-syscall addressing).
- `src/host/storage-engine/index.ts` — `createStorageEngine`, `deleteStorage` exports.
- `src/host/storage-engine/bindings/op-sqlite.ts` — device file naming (`storage/<appId>.db`).

- `src/host/storage-engine/schema.ts` — `validateArtifact`, `diffSchemas`, `AppliedSchema`/`SchemaDiff`.
- `openspec/specs/mini-app-storage/spec.md`, `openspec/specs/storage-schema-evolution/spec.md` — governing requirements, one of which this change must supersede.

## Current behavior
**1. Engine appId mapping (host-side only, never bundle-chosen).** `StoreAccess.engineAppId(entry)` returns `entry.id` unconditionally (`store-access.ts:70-72`) — the launcher id, always. `useMiniAppHost.bind()` receives this as `engineAppId` and calls `launchApp({...record, appId: engineAppId}, createEngine, generation)` (`useMiniAppHost.ts:123-127`), overriding the bundle's own record `appId`. `launchApp` (`launch.ts:30,41`) calls `createEngine(app.appId)` — i.e. `createStorageEngine({appId: engineAppId})` — then `engine.open(app.schemaArtifact)` (`launch.ts:43`) before the realm is considered launched; a conflict-class open failure aborts the launch (`launch.ts:44-54`) and the bundle never runs. The device binding resolves the appId to a file: `open({name: '${appId}.db', location: 'storage'})` (`bindings/op-sqlite.ts:31`). The bundle itself never supplies or sees this id — it is entirely host-injected at bind time (`useMiniAppHost.ts:106-127` comment: "the engine is opened under the LAUNCHER id, not the bundle's appId").

**2. Delete path.** `StoreAccess.remove(entry)` (`store-access.ts:143-151`): removes the index entry (line 145), then calls `this.deleteStorage(entry.id)` **unconditionally** (line 146, comment "no per-app user-data residue (D8)") — no refcount check on the engine appId. Only the version-store repo removal is refcounted: `if (this.index.refCount(repo) === 0) await this.store.remove(repo)` (lines 147-148). `deleteStorage({appId})` (`storage-engine/index.ts:42-46`) opens `${appId}.db` via op-sqlite and calls `db.delete()` — deletes the whole file. `AppIndex.refCount(storeId)` (`app-index.ts:108-110`) counts installed entries whose `(a.storeId ?? a.id) === storeId` — this is the exact pattern a refcounted storage delete would need to mirror, but keyed on whatever field carries the shared engine-appId instead of `storeId`.

**3. InstalledApp shape / no existing storage indirection.** `InstalledApp` (`app-index.ts:24-41`) has `id`, `storeId?` (version-store repo, forks only), `lineageId`, `forkedFrom?` — nothing analogous for storage. `engineAppId()` is a pure function of `entry.id` (`store-access.ts:70-72`); there is no field today that could carry a "storage group" id distinct from the launcher id. Introducing sharing needs a new field (e.g. an optional storage-group id) plus a new `engineAppId()` branch.

**4/5. Schema validation, diffing, and ID allocation.** `validateArtifact(artifact): StorageError[]` (`schema.ts:170`, exported at `storage-engine/index.ts:20`) checks **artifact-internal validity only**: shape, reserved names ("id"), `BURNED_ID_RE` format, in-artifact ID reuse, active/tombstone contradiction, closed-type-set membership, default validity (`schema.ts:98-198`) — it has **no knowledge of any accumulated/applied schema**. `diffSchemas(applied: AppliedSchema, incoming: SchemaArtifact): SchemaDiff` (`schema.ts:313`, exported) is the piece that checks a *new* artifact against `_meta`'s accumulated union: it already detects "same field ID, different type" as a `type_change` conflict (`schema.ts:246-248`) and "same ID, different meaning after tombstone" as `tombstone_violation` (`schema.ts:252-260`) — **this pre-existing cross-generation conflict logic is exactly the fail-closed collision guard the linked-apps change needs**; it is keyed purely on burned ID vs. `_meta`, with no notion of *which app* produced an artifact, so it would fire identically whether the colliding artifact came from a later generation of the same app or from a second app sharing the DB. `launchApp` calls `engine.open(schemaArtifact)` (`launch.ts:43`), which runs `diffSchemas` internally (not shown here) and throws `StorageEngineError` on conflict, aborting the launch (`launch.ts:44-54`) — so a fail-closed collision on shared storage would already surface as a structured launch error via the existing path, not silent data corruption.
**ID allocation**: `contract.ts:69-71` comments IDs as "engine-minted from `[a-z]\d+`," but I found **no runtime ID-minting function** anywhere in `src/host/storage-engine/` — `validateArtifact` only *checks format* (`BURNED_ID_RE.test`, `schema.ts:113,131,150`), it never generates an ID. Every artifact arrives with IDs already burned in it. Decision #38 (`docs/decisions.md:363`) says "every persisted field gets a stable ID in the declared-schema artifact," consistent with IDs being assigned wherever the schema artifact is authored (today: hand-authored fixtures/`device-acceptance.ts`; in product terms, the future generation harness, #11, unproposed) — **allocation today is artifact-author-side, not engine-side**, despite the "engine-minted" comment language.

**6. Fork flow UI.** `HomeScreen`'s long-press action sheet has an `onFork` row (`HomeScreen.tsx:132`) → `LauncherRoot.onFork` calls `access.fork(app)` (`LauncherRoot.tsx:104-111`) → `StoreAccess.fork(entry)` (`store-access.ts:116-135`) creates a new `InstalledApp` with a fresh `id` (`${repo}__${lineageId}`) and no `record`/schema modification — it copies `entry.record` verbatim (`store-access.ts:129`). There is currently **no branch point** in this flow for a share-vs-fresh storage question; it would need to be inserted between the long-press "Fork" tap and `access.fork()` (e.g. an intermediate prompt/sheet), and `StoreAccess.fork` would need a new parameter threaded to whatever creates/reuses the storage-engine binding.

## Constraints and invariants
- **mini-app-storage spec, Requirement 1** (`openspec/specs/mini-app-storage/spec.md:6-8`): "Each mini-app's user data SHALL live in its own physically separate SQLite database (one database file per app)... the engine API MUST NOT accept any per-call app addressing." This change's premise (two launcher entries sharing one DB file/engine instance) requires a spec delta here — sharing does not violate "no per-call addressing" (each engine instance is still bound to exactly one appId at construction) but does contradict "one database file per app" read at the launcher-entry granularity; the spec's "mini-app" unit of isolation needs redefining (per-lineage-family vs. per-launcher-entry).
- **storage-schema-evolution spec** (`openspec/specs/storage-schema-evolution/spec.md`): additive-only evolution, burned-ID-never-reused, tombstone-on-delete — all enforced by `diffSchemas`/`validateArtifact` against `_meta`'s accumulated union; these rules are exactly what a shared-DB collision guard would lean on, unmodified.
- **Decision #43b D2/D8** (`docs/decisions.md:495`): "the runtime ENGINE appId is always the launcher id, so a fork gets its own user data even sharing a repo" — this is the exact rule the new change supersedes; must be recorded as a reversal in the decisions log, not silently overwritten.
- **Capability-bridge D2** (`docs/decisions.md` #41, `capability-bridge/spec.md`): the syscall envelope carries no app/store/realm field (`contract.ts` `RealmRecord`, no `appId` in per-syscall frames) — authority is channel-derived (which WebView/dispatcher closure, bound at launch). Sharing storage must not add an appId parameter anywhere in the syscall surface; it can only change what the host binds a realm's engine handle *to* at `launchApp` time — a zero-transport-diff change, matching the #41 "capability addition = registry row + client stub only" review rule (not applicable here since no new capability is added, but the "don't touch transport/dispatcher" discipline still applies).
- `launchApp`'s engine-then-schema-open-before-delivery sequencing (`launch.ts:30-65`) is where any new share-vs-fresh engine-resolution logic would have to sit, since it's the single choke point that already turns a schema conflict into a pre-delivery structured failure.

## Integration points
- `StoreAccess.engineAppId` (`store-access.ts:70-72`) — needs a share-aware branch (today: pure function of `entry.id`).
- `StoreAccess.fork` (`store-access.ts:116-135`) and `HomeScreen`'s action sheet (`HomeScreen.tsx:126-137`) → `LauncherRoot.onFork` (`LauncherRoot.tsx:104-111`) — where a share-vs-fresh prompt would be asked and threaded through.
- `StoreAccess.remove` (`store-access.ts:143-151`) — `deleteStorage` call needs to become refcounted, mirroring `AppIndex.refCount(storeId)` (`app-index.ts:108-110`) but keyed on the storage-sharing id.
- `InstalledApp` (`app-index.ts:24-41`) — needs a new field to carry the shared engine-appId/storage-group, parallel to how `storeId` already carries the shared version-store repo.
- `launchApp` (`launch.ts:30-65`) — the engine-creation choke point; `diffSchemas`/`validateArtifact` (`schema.ts:170,313`) are the pre-existing fail-closed collision machinery to be reused, not rebuilt.
- `openspec/specs/mini-app-storage/spec.md` Requirement 1 — needs a spec delta to redefine the isolation unit.

## Risks and unknowns
- **Cross-app SQLite concurrency**: one WebView == one realm == one app is the standing invariant (`useMiniAppHost.ts:6-8,58`; decision #41 D2) — only one realm is ever live at a time, so two launcher entries sharing a DB file are never opened concurrently by two live engine instances today. I did not verify what happens to an already-`open()`ed op-sqlite handle when the *previous* realm's engine is closed (`bind()` calls `live.current.realm.engine?.close()` at `useMiniAppHost.ts:111`) versus a *fresh* realm reopening the same file moments later — this ordering (close-then-reopen, always serialized by the realm-bind sequence) is probably safe by construction but I did not trace op-sqlite's file-handle lifecycle to confirm no residual lock.
- I did not verify where/how the (currently nonexistent) ID-minting will eventually happen in the unproposed `#11 generation-loop` change — the "engine-minted" comment in `contract.ts:69` may reflect forward-looking intent rather than current or planned architecture; this is worth flagging to the proposer rather than assuming.
- I did not check `src/host/bridge/test/acceptance.ts:289`'s existing conflict-schema test in detail (`launchApp` with a `conflict` artifact against a shared engine) — it may already partially exercise the exact "two schemas, one engine" scenario this change formalizes; worth a closer read before designing new tests.
- The exact wording of the future spec delta for `mini-app-storage` Requirement 1 (what unit "isolation" now applies to) is a judgment call, not derivable from code.
