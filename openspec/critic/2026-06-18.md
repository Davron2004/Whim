## 🔴 High-severity findings (15) — read these first

### A1 · loader.js message listener accepts \_\_whimDeliver from any source, not just window.parent

- **Area:** A — Sandbox runtime + SDK · **Category:** `broken-invariant`
- **Location:** `src/runtime/web/loader.js:110-185`
- **Problem:** The window.addEventListener('message', ...) handler at line 110 does not check ev.source. syscall.js performs this check correctly (line 69: if (ev.source !== window.parent) return). A malicious bundle can call window.postMessage(JSON.stringify({`__whimDeliver`: true, bundle: 'arbitrary code'}), '\*') asynchronously (e.g. from a setTimeout that fires after deliveryBusy resets at line 185). The loader will process this as if it were a host-sent delivery: it increments `__whimGeneration`, wraps and injects the bundle source as an inline script, and posts an authenticated 'delivery' frame (with the real hostNonce) to the host. The injected code still runs under the same CSP and neutralized globals so no containment escape is achieved, but the host's state machine receives a generation-counter increment and a delivery-accepted frame it never initiated. The same path allows unauthorized re-injection without the host's knowledge, violating the design intent that the host controls all bundle delivery. The asymmetry with syscall.js (which does the ev.source check) makes this look like an unintentional omission.
- **Recommendation:** Add if (ev.source !== window.parent) return; as the first guard in the loader's message handler at line 111, mirroring syscall.js line 69. The 'hello' frame posted at line 192 flows the other direction (iframe to host) and is unaffected. Add a probe to probes.js T6 that verifies self-posted \_\_whimDeliver frames are ignored.

### C1 · Compaction failure inside snapshot() makes persisted snapshot appear to fail

- **Area:** C — Version store · **Category:** `broken-invariant`
- **Location:** `src/host/version-store/engine.ts:208-221`
- **Problem:** snapshot() commits and tags the new snapshot at lines 208-216, then conditionally calls compactRepo() at line 219 with no try/catch. If compactRepo throws (e.g., packObjects or indexPack fails), the exception bubbles out of snapshot(), so the caller sees a rejection. The snapshot commit and its snap tag are already durably written in the FS/KV backend at that point. The caller cannot distinguish 'snapshot was not saved' from 'snapshot was saved but compaction failed', and a naive retry would create a second commit for the same generation with a new ID. For a KvBackedFs store this means the KV keys persist with the committed data but the application believes the operation failed.
- **Recommendation:** Wrap the compactRepo call in a try/catch and record the compaction failure separately without re-throwing — the snapshot itself has succeeded. Alternatively, move auto-compaction to a fire-and-forget path triggered after snapshot() returns successfully. Either way, snapshot() should have a clean all-or-nothing contract visible to the caller.

### C2 · Orphaned loose objects from partial snapshot() never compacted — slow KV key leak

- **Area:** C — Version store · **Category:** `data-loss`
- **Location:** `src/host/version-store/engine.ts:194-206, src/host/version-store/compaction.ts:90-93`
- **Problem:** snapshot() calls git.add() for each artifact (writing loose blob objects to the FS) before git.commit(). If anything throws between the first git.add and git.commit (e.g., writeFile fails, isomorphic-git throws, or the process is killed), loose blob objects are written to the FS/KV backend but no commit references them. compactRepo()'s deletion loop at compaction.ts:91 guards `if (reachable.has(oid))` and skips objects not reachable from any ref. Orphaned blobs from a failed snapshot are therefore never deleted — they accumulate indefinitely in KV, partially defeating the key-count pressure argument that justifies the compaction design.
- **Recommendation:** After compaction packs reachable objects, add a second pass that also unlinks loose objects NOT in `reachable` (true garbage collection). Since these objects are by definition unreferenced, deleting them is safe. Alternatively, document the known leak and add a measurement to the device acceptance to surface its growth rate.

### D1 · Field with display name 'id' silently clobbers the engine-assigned rowid in list() output

- **Area:** D — Storage engine · **Category:** `bug`
- **Location:** `src/host/storage-engine/engine.ts:145-146, src/host/storage-engine/schema.ts:96-165`
- **Problem:** records.list() builds the output object in two steps: line 145 sets `out = { id: Number(row[RECORD_ID]) }` (the engine-assigned integer primary key), then line 146 iterates all active fields and sets `out[name] = fromStorage(...)`. If any field has the display name 'id', the second step overwrites the rowid with that field's value. validateArtifact() checks burned IDs against BURNED_ID_RE but never validates display names — 'id' passes all checks cleanly. The result is a StorageRecord where the `.id` property holds the wrong value (or the wrong type), breaking every downstream caller that relies on the id for update/remove. The BURNED_ID_RE alphabet (`[a-z][0-9]+`) ensures burned IDs cannot be 'id' (no digit), so quoteIdent(RECORD_ID) is safe — but the display-name collision in the JS output layer is unguarded.
- **Recommendation:** Add a check in validateArtifact() that rejects any field whose display name equals the reserved string 'id' (and possibly any collection whose display name equals 'id'). Alternatively, rename the output key to something like '\_id' or 'rowid' and update the StorageRecord type — but that is a larger seam break since the contract is imported by capability-bridge.

### D2 · 'unknown_record' declared in StorageErrorKind but never thrown — contract promises behavior the engine does not implement

- **Area:** D — Storage engine · **Category:** `broken-invariant`
- **Location:** `src/host/storage-engine/contract.ts:161, src/host/storage-engine/engine.ts:151-169`
- **Problem:** The contract's StorageErrorKind union includes 'unknown_record', which implies that update() and remove() raise a structured error when called with a non-existent id. Neither verb does: update() runs the UPDATE and silently returns if zero rows were changed; remove() runs DELETE with no rowcount check. The test suite has no test for this error kind. Any caller that assumes `update(collection, staleId, patch)` throws on a missing record will silently corrupt state — they'll never know the operation had no effect. The bridge layer (decision #41) dispatches engine errors verbatim to the mini-app; a silent no-op cannot be retried or surfaced.
- **Recommendation:** Either: (a) implement the guard — after the UPDATE/DELETE, check `res.changes === 0` and throw `storageError({ kind: 'unknown_record', ... })`, then add a test for this path; or (b) remove 'unknown_record' from StorageErrorKind and update the spec if silent no-op is the deliberate design. The first option matches what the contract promises; option (b) requires the spec to document the silent-no-op contract.

### E1 · DevProbeScreen '‹ Home' button bypasses host.exit(), leaking engine handle and live realm

- **Area:** E — Host shell, capability bridge, launcher & RN root · **Category:** `broken-invariant`
- **Location:** `src/host/launcher/DevProbeScreen.tsx:40`
- **Problem:** The back button `onPress={onExit}` calls the passed `onExit` prop directly (which is `LauncherRoot.goHome`), not `host.exit()`. `host.exit()` is the ONLY path that calls `tearDownRealm(live.current.realm)` and `live.current.realm.engine?.close()`. When the DevProbeScreen is navigated away via this button, the live realm's `alive` flag stays `true`, the op-sqlite DB handle is never closed, and any in-flight async syscall could complete against a now-orphaned dispatcher and attempt to call `control()` into a WebView that no longer exists. The app-launcher spec and the invariants in useMiniAppHost both require teardown on leave; this path silently skips it. MiniAppView does not share this bug (FloatingExit calls `host.exit()` before returning control to the parent).
- **Recommendation:** Change the DevProbeScreen back button to `onPress={host.exit}` and pass `onExit` as the `onExit` option to `useMiniAppHost`. Alternatively, add a `useEffect` cleanup in `useMiniAppHost` that calls the equivalent of `exit()` on unmount, covering any navigation path that bypasses `host.exit()`.

### E2 · useMiniAppHost has no unmount cleanup for the live realm — teardown invariant depends entirely on callers

- **Area:** E — Host shell, capability bridge, launcher & RN root · **Category:** `broken-invariant`
- **Location:** `src/host/launcher/useMiniAppHost.ts:235-252`
- **Problem:** The only `useEffect` in `useMiniAppHost` is the `BackHandler` registration, whose cleanup only removes the listener and clears `popTimer`. There is no cleanup effect that calls `tearDownRealm` + `engine.close()` on unmount. The current product code relies on callers to call `host.exit()` before their parent unmounts them. This is fragile: any new screen that uses `useMiniAppHost` must remember to route all its exits through `host.exit()` or it leaks an open DB handle and a live realm. The DevProbeScreen bug (E1) is one concrete expression of this structural gap. A React devtools Strict Mode double-mount would also expose it.
- **Recommendation:** Add a `useEffect(() => { return () => { /* call exit() logic without the onExit callback */ }; }, [])` in `useMiniAppHost` that closes the engine and tears down the realm on unmount. This makes the invariant self-enforcing regardless of the caller's exit path.

### H3 · typecheck, lint, knip, openspec validate, and scaffolding tripwires are absent from CI

The overall posture is good for a first baseline, but findings A1 and A2 are real gaps on the containment-critical path.

#### 🔴 HIGH · `broken-invariant` · C1 — Compaction failure inside snapshot() makes persisted snapshot appear to fail

- **Location:** `src/host/version-store/engine.ts:208-221`
- **Problem:** snapshot() commits and tags the new snapshot at lines 208-216, then conditionally calls compactRepo() at line 219 with no try/catch. If compactRepo throws (e.g., packObjects or indexPack fails), the exception bubbles out of snapshot(), so the caller sees a rejection. The snapshot commit and its snap tag are already durably written in the FS/KV backend at that point. The caller cannot distinguish 'snapshot was not saved' from 'snapshot was saved but compaction failed', and a naive retry would create a second commit for the same generation with a new ID. For a KvBackedFs store this means the KV keys persist with the committed data but the application believes the operation failed.
- **Recommendation:** Wrap the compactRepo call in a try/catch and record the compaction failure separately without re-throwing — the snapshot itself has succeeded. Alternatively, move auto-compaction to a fire-and-forget path triggered after snapshot() returns successfully. Either way, snapshot() should have a clean all-or-nothing contract visible to the caller.

#### 🔴 HIGH · `data-loss` · C2 — Orphaned loose objects from partial snapshot() never compacted — slow KV key leak

- **Location:** `src/host/version-store/engine.ts:194-206, src/host/version-store/compaction.ts:90-93`
- **Problem:** snapshot() calls git.add() for each artifact (writing loose blob objects to the FS) before git.commit(). If anything throws between the first git.add and git.commit (e.g., writeFile fails, isomorphic-git throws, or the process is killed), loose blob objects are written to the FS/KV backend but no commit references them. compactRepo()'s deletion loop at compaction.ts:91 guards `if (reachable.has(oid))` and skips objects not reachable from any ref. Orphaned blobs from a failed snapshot are therefore never deleted — they accumulate indefinitely in KV, partially defeating the key-count pressure argument that justifies the compaction design.
- **Recommendation:** After compaction packs reachable objects, add a second pass that also unlinks loose objects NOT in `reachable` (true garbage collection). Since these objects are by definition unreferenced, deleting them is safe. Alternatively, document the known leak and add a measurement to the device acceptance to surface its growth rate.

#### 🟠 MED · `bug` · C3 — Old pack files accumulate with each compaction cycle, leaking KV keys

- **Location:** `src/host/version-store/compaction.ts:84-94`
- **Problem:** Each compactRepo() call calls packObjects() for all reachable objects, which creates a NEW .pack and .idx file in objects/pack/. Alternatively, rename the output key to something like '\_id' or 'rowid' and update the StorageRecord type — but that is a larger seam break since the contract is imported by capability-bridge.
