### [severity: high] `invariants/` is hook-unprotected outside a fix-loop worktree — the very PR that hardened the worktree case left the main-tree case as-is
- Where: `.claude/hooks/protect-harness.sh:99-108` (main-tree block-list, unchanged by this range) vs `.claude/hooks/protect-harness.sh:74-97` (new in-worktree Class-2 block, added in 8b503ae) and `.claude/hooks/bash-policy.sh:63` (PROTECTED regex, also missing `invariants`)
- Category: 4 (structural smell) / 3 (spec drift vs CLAUDE.md's "invariants are authored by runtime owners, never a feature-implementing agent")
- What: 8b503ae rebuilt `protect-harness.sh`'s in-worktree carve-out specifically to close "the previous blanket carve-out silently allowed a subagent to edit gate.sh/the fix-loop script/invariants/ in-worktree" (the file's own comment, line 67), and correctly added `*/.claude/worktrees/*/invariants/*` to the new hard Class-2 block (line 79). But the pre-existing, untouched main-tree block-list at lines 99-108 (`scripts/gate.sh`, `.claude/*`, eslint/knip/tsconfig/package.json, `build/*`, …) still has no `invariants/*` entry, and `bash-policy.sh`'s `PROTECTED` write regex (line 63) never mentions `invariants` either. A subagent editing `invariants/**` directly in the main tree (the normal mode for a feature-chain implementer per the dispatcher model — not every subagent runs under `isolation: worktree`) hits neither hook and falls through to `exit 0` — unrestricted Edit/Write.
- Why it matters: `invariants/` is the never-regress sandbox-isolation suite — CLAUDE.md is explicit that it "is owned by runtime owners; a feature chain editing it is a report-worthy boundary violation," and the whole current PR is about mechanizing exactly this boundary. Today enforcement is convention-only in the one context (main tree) most subagents actually run in; a feature-implementing subagent that "fixes" a failing invariant by editing the invariant instead of the code would be caught only after the fact by a reviewer reading the diff, not prevented.
- Suggested approach: add `*/invariants/*|invariants/*` to protect-harness.sh's main-tree case block (mirroring the `build/*` entry right next to it) and to bash-policy.sh's `PROTECTED` regex. Not mechanical-lane eligible — it's an edit to `.claude/hooks/**`, which is Class-2 and human/main-thread-only under the very model this diff introduced.

### [severity: high] bash-policy's write-verb boundary regex has an unguarded subshell gap — `(cp x package.json)` bypasses the protected-write hard-deny
- Where: `.claude/hooks/bash-policy.sh:58-67` (`BND='(^|[[:space:]&;|])'`), landed by aeccc05, whose entire point was anchoring write-verbs at a command boundary
- Category: 4 (structural smell) — a security-control regression hiding inside its own fix
- What: `BND` treats `^`, whitespace, `&`, `;`, `|` as boundaries but not `(`. A bash subshell needs no space after the paren (`(cp evil.txt package.json)` is valid bash), so `cp` there is preceded by `(`, which is in none of the boundary classes — the write-protection `grep -Eq` does not match. Separately, the "compound command falls through to normal permission flow" safety net at line 73-76 lists `&`,`;`,`|`, backtick, `$(`, `<`, `>`, newline as compound indicators but not a bare `(` — so this command is also *not* flagged as compound and proceeds straight to the git-policy / auto-allow sections. Verified live: `(cp evil.txt package.json)` matches neither the protected-write regex nor the compound-fallthrough case (tested by extracting the exact regex/BND from the file and running it against that string). The same gap applies to every other verb in that alternation (`sed -i`, `tee`, `mv`, `install`, `dd`, `truncate`) and to any of bash-policy's other `PROTECTED` targets (`scripts/gate.sh`, `.claude/hooks`, etc.), not just `package.json`.
- Why it matters: this is the exact class of bug aeccc05 was fixing (boundary-anchoring false negatives/positives), and it left one boundary character out. It's a hard-coded, deterministic control that the harness design explicitly trusts as "the real backstop" alongside the gate's BASE-diff tripwire — a silent bypass here means a subagent (or a corrupted prompt) can mutate protected config via a trivially-shaped command and only the (softer, best-effort) integrity/gate-based tripwires would catch it after the fact, if at all outside the fix-loop flow.
- Suggested approach: add `(` (and, for symmetry, `{`) to `BND`, or — more robustly — stop relying on a single-line boundary-char class and instead reject any command containing a `(` before falling through, the way `$(` already is. Add a regression case for `(cp|sed|tee|mv) ... <protected>` to whatever unit-tests bash-policy.sh (the aeccc05 commit message claims "12/12 hook unit cases green" — this shape isn't one of them).

### [severity: med] bash-policy's fixloop-orchestrator-only guard denies pure reads too, contradicting its own "reads are untouched" policy
- Where: `.claude/hooks/bash-policy.sh:29-32` vs the comment at lines 52-54 ("Reads (cat/grep/git diff) are untouched; the legit edit path is the Edit tool")
- Category: 4 (structural smell)
- What: the guard (the exact literal path substring appears at that line) is a match-anywhere check with no verb discrimination — it fires for `git diff … -- <that path>`, `git log -p -- <that path>`, `git show <ref>:<that path>`, `cat <that path>`, `grep … <that path>`, not just for actually *executing* the script. This is demonstrably live: this critic run hit exactly this deny on a plain read-only `git diff` naming that path, and again — twice — when writing THIS report, because the report's own prose discusses the file by name.
- Why it matters: the design's stated intent ("reads are untouched") is violated for exactly one file, for exactly the audience (read-only auditing subagents — critics, reviewers) that most needs unimpeded read access to a script explicitly documented as security-relevant ("git run *inside* this script does not re-trigger the PreToolUse(Bash) hook"). It doesn't block a determined agent (a trivial pathspec-glob or path-without-prefix workaround exists) so it isn't a security hole, but it silently degrades every future read-only review of this file — or any file/report that quotes its path — and is inconsistent with how every other protected path in the same script is handled (write-only blocking).
- Suggested approach: narrow the case pattern to command shapes that actually invoke the script (a leading interpreter/relative-path invocation naming it, optionally as the first token of a chain) instead of match-anywhere on the bare path string.

### [severity: med, category: 5 REAL-BUT-HARD] the new worktree-`cd` auto-allow isn't scoped to the calling agent's own worktree
- Where: `.claude/hooks/bash-policy.sh:101-106` (new, §6.9) combined with the pre-existing CWD-scoped git policy at `.claude/hooks/bash-policy.sh:82-99`
- Category: 5 (real-but-hard) / 4 (structural smell)
- What: the new rule auto-allows any simple `cd …/.claude/worktrees/…` command (rejecting only ones containing `..`) so a fix-worker launched without a pre-created worktree can `cd` into the one the orchestrator names. But the glob doesn't check that the named worktree id is the one *this* agent invocation was actually given — it matches `cd` into any sibling worktree path too. The git policy right above it (lines 86-94) grants `add`/`commit`/`checkout`/`switch`/`restore`/`stash`/`branch`/`rev-parse` to a subagent whenever `$CWD` is merely "somewhere under `.claude/worktrees/*`," not "under *my* worktree." `ls` is separately auto-allowed, so a worker can enumerate `.claude/worktrees/` to discover sibling ids. Chained together: a fix-worker subagent can discover, `cd` into, and run mutating git against a *different* fix-worker's isolated worktree.
- Why it matters: the entire parallel fix-loop's safety argument rests on "N parallel fixers, isolated worktrees, one writer to dev/v1 at a time" (docs/parallel-fix-loop.md). A worker able to reach into a sibling's tree — even accidentally, e.g. a confused agent that mis-copies a path from context — breaks that isolation assumption in a way none of the later checks (integrity/redcheck/gate) are designed to catch, since they diff a *branch* against its BASE, not "did anything else touch this worktree mid-flight."
- Why it's hard: flagged as REAL-BUT-HARD deliberately — the hook is a stateless shell script that sees only `$CMD`/`$CWD`/`$AGENT_ID` per call; it has no durable notion of "which worktree id was this particular agent invocation assigned." Closing it cleanly needs either passing/reading an agent-to-worktree binding (e.g. a per-agent grant-style manifest analogous to the existing grants directory) or trusting the orchestrator never to leak sibling paths into a worker's context — a policy discipline, not a mechanical check.
- Suggested approach (sketch only): write a small agent-unwritable ownership marker (same trust model as the Class-1 grants) when the orchestrator creates or hands off a worktree, and have the hook check `$AGENT_ID` against it before allowing `cd`/scoped-git into that id — otherwise fall through to the normal (stall/ask) path.

## Patterns worth a tripwire

- **Boundary-anchored regexes need a `(` case too.** `BND='(^|[[:space:]&;|])'` in bash-policy.sh (and any future copy of this pattern) should include `(` — a grep-able tripwire: `grep -n "BND=" .claude/hooks/bash-policy.sh` and manually confirm `(` is in the class whenever this file is touched.
- **Any new hook case block that lists specific protected directories should be diffed against its sibling block.** `protect-harness.sh` now has two independent lists of "protected paths" (main-tree case and in-worktree Class-2 case) that must stay in lockstep by hand — a consistency check (or refactoring both off one shared array, mirroring how the fixloop toolkit already keys off CLASS1/CLASS2 bash arrays) would catch drift like the `invariants/` gap mechanically instead of relying on a human noticing.
- **A match-anywhere deny keyed to one specific file's literal path is a smell** — every other protected-file rule in this codebase is scoped to write verbs; a lone read-blocking exception for a single path is easy to introduce by copy-paste and easy to forget is inconsistent with the stated design (and, concretely, trips on any prose — including a critic report — that quotes the path).
